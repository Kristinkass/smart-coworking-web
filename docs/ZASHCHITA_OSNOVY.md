# Как у нас всё работает — шпаргалка для защиты

Простыми словами: что за система, через что идут запросы, где лежат данные и что кэшируется.

---

## 1. Что это за система

**Умный коворкинг** — веб-приложение для управления пространством:

- интерактивная **карта** мест (столы, переговорные, зоны);
- **бронирование** по времени (шаг 15 минут, минимум 30 минут);
- **тарифы** (почасовой, недельный, месячный) и **абонементы**;
- **роли**: клиент, менеджер, администратор;
- **админка**: пользователи, отчёты, расписание, уведомления, редактор планировки.

Пользователь открывает сайт в браузере — отдельное мобильное приложение не нужно.

---

## 2. Технологии (стек)

| Слой | Что используем |
|------|----------------|
| Сервер | **Python 3**, фреймворк **Flask** |
| База данных | **PostgreSQL** |
| ORM | **SQLAlchemy** (модели в `internal/models/`) |
| Авторизация | **Flask-Login** (сессия в cookie) |
| Шаблоны страниц | **Jinja2** (`templates/`) |
| Фронтенд | HTML, CSS, JavaScript (без React/Vue) |
| Карта | **SVG** + JS (`static/js/map_updated.js`) |
| Продакшен | **Waitress** + **Docker** |

Точка входа: `wsgi.py` (прод) или `cmd/app/main.py` (разработка).

---

## 3. Архитектура — «кто за что отвечает»

Запрос от браузера проходит так:

```
Браузер (HTML + JS)
    ↓ HTTP (страница или /api/...)
Flask — handlers (internal/handlers/)
    ↓ вызывает
Сервисы (internal/services/) — бизнес-логика
    ↓ через
Репозитории (internal/repositories/) — запросы к БД
    ↓
PostgreSQL
```

Отдельно для карты:

```
layout.json (геометрия)  ←→  sync по code  ←→  PostgreSQL (места, брони)
```

**Handlers** — только маршруты: принять запрос, проверить права, вернуть HTML или JSON.  
**Services** — правила: можно ли забронировать, цена, сетка времени, пересечения столов.  
**Models** — таблицы в БД: пользователи, брони, тарифы, расписание.

Регистрация всех маршрутов: `internal/handlers/__init__.py` → `register_all_handlers(app)`.

## 4. Роли пользователей

| Роль | Вход | Основное |
|------|------|----------|
| **client** | Телефон (основной) или email | Карта `/mapp`, своё бронирование, личный кабинет `/dashboard` |
| **manager** | Как клиент + права сотрудника | Бронь за клиента, клиенты, часть админки |
| **admin** | Как выше | Всё: редактор планировки, тарифы, отчёты, пользователи |

Проверки в коде: `@login_required`, `@staff_required`, `@admin_required` (`internal/utils/decorators.py`).

Пароли в БД **не хранятся открытым текстом** — только хеш (Werkzeug).

## 6. Сценарий: как работает бронирование

1. Пользователь открывает **`/mapp`** — сервер отдаёт HTML (`templates/mapp.html`).
2. JS запрашивает **`GET /api/places`** — список мест, статусы (свободно / занято / частично), стены, тарифы.
3. Пользователь выбирает место и дату.
4. JS запрашивает **`GET /api/booking/timegrid/<place_id>?date=...`** — сетка 15-минутных слотов (что занято, что свободно).
5. При нажатии «Забронировать» — **`POST /api/booking/create`** (или абонементный API).
6. Сервер в `booking_service`:
   - смотрит расписание коворкинга на этот день;
   - проверяет пересечения с другими бронями;
   - считает цену по тарифу;
   - пишет запись в таблицу `bookings` в PostgreSQL.
7. Карта обновляется повторным запросом `/api/places`.

**Важно:** минимальный интервал — **30 минут**, шаг сетки — **15 минут**.

Менеджер может выбрать клиента в форме — бронь создаётся на него (`user_id` в запросе).

---

## 7. Сценарий: редактор планировки

1. Админ открывает **`/editor`**.
2. План читается из **`static/layout.json`** (координаты столов, стены, двери).
3. Перетаскивание стола → **`POST /api/admin/place/move`** (и похожие API).
4. Сервер проверяет геометрию (не на стене, не на другом столе) — `internal/layout/geometry.py`.
5. Успех → JSON на диск **атомарно** перезаписывает `layout.json`, вызывается **`reload_layout()`** — сброс кэша в памяти.
6. Метаданные места (название, тариф, вместимость) — в **PostgreSQL**, связь с layout по полю **`code`** (например `1A-T5`).

## 8. Где что хранится (не путать с кэшем)

```
┌─────────────────────────────────────────────┐
│  PostgreSQL                                 │
│  • users, bookings, tariffs, subscriptions  │
│  • places (id, code, capacity, status…)     │
│  • schedule (часы работы по дням недели)    │
│  • notifications, ratings                   │
│  • НЕ хранит x, y, width, height столов     │
└─────────────────────────────────────────────┘
              ↕ связь по code места
┌─────────────────────────────────────────────┐
│  static/layout.json                         │
│  • координаты, поворот, стены, двери        │
│  • источник правды для отрисовки карты      │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  static/js, static/css, templates/          │
│  • фронтенд, подключается браузером         │
└─────────────────────────────────────────────┘
```

Бронь привязана к **`place_id`** в БД. Смена кода места (`1A-T5` → `1A-T3`) **не ломает** старые брони.

## 9. Что кэшируется (три разных вещи)

### 9.1. Кэш браузера (самый частый «баг» на демо)

Браузер сохраняет старые **JS и CSS**, чтобы не качать каждый раз.

- **Где:** на компьютере / телефоне пользователя.
- **Симптом:** «время съехало», старые кнопки, не обновилась карта.
- **Как боремся:** в шаблонах версия в URL: `map_updated.js?v=32`. Изменили файл → **увеличили `v`** в `mapp.html` / `base.html`.
- **Как обновить у себя:** `Ctrl+F5` (Windows) или закрыть вкладку на телефоне и открыть заново.

Это **не баг сервера** — это кэш клиента.

### 9.2. Кэш планировки в памяти сервера

Файл `layout.json` после первого чтения держится в переменной **`_LAYOUT_CACHE`** (`internal/layout/store.py`).

- **Где:** в процессе Python (Flask / Waitress).
- **Зачем:** не читать диск на каждый запрос карты.
- **Когда сбрасывается:**
  - `reload_layout()` — после сохранения через редактор/API;
  - перезапуск контейнера / сервера.
- **Симптом:** на диске layout новый, на карте старые координаты — не вызвали `reload_layout` или не перезапустили после ручного редактирования файла.

### 9.3. Кэш синхронизации layout ↔ БД

В `internal/layout/sync.py` есть вспомогательный кэш сопоставления мест — сбрасывается вместе с `reload_layout()` / `reset_sync_cache()`.

## 10. Что НЕ кэшируется (и это нормально)

| Данные | Поведение |
|--------|-----------|
| Бронирования | Всегда из PostgreSQL при каждом API-запросе |
| Статусы мест на карте | Считаются при `GET /api/places` по актуальным броням |
| Timegrid (слоты) | Каждый запрос `/api/booking/timegrid/...` — свежие данные с сервера |
| Сессия пользователя | Cookie + Flask-Login, не «кэш броней» |

**WebSocket нет.** Обновление карты и шкалы времени на клиенте — через **повторные HTTP-запросы** (после брони и по таймеру ~15 с, пока открыта форма бронирования).

## 11. Время и часовой пояс

| Вопрос | Ответ |
|--------|--------|
| Какой TZ у сервера? | `Europe/Moscow` (`APP_TIMEZONE` в config / Docker) |
| «Сегодня» на карте | Ставит **браузер** (`initBookingDate()` в JS) |
| Прошлые слоты | Помечает и клиент, и сервер в timegrid |
| Почему не UTC в API? | Бронь — по «настенным часам» коворкинга, не по поясу пользователя |

Если на защите спросят про «+3 часа» — ответ: проверить TZ контейнера и жёсткое обновление JS (`?v=`).

## 13. Безопасность — коротко для комиссии

- Пароли — **хеш**, не plaintext.
- Доступ к API бронирования и админке — только **после входа**.
- Редактор и движение мест — только **admin**.
- Формы — **CSRF** (Flask-WTF).
- Геометрия столов проверяется **и в браузере, и на сервере** (нельзя обойти только JS).
- На проде обязательны: свой **`SECRET_KEY`**, **`DEBUG=False`**, HTTPS.

Подробнее про риски: `docs/COMMISSION_QA.md`.

## 14. Деплой и когда что перезапускать

| Изменили | Действие |
|----------|----------|
| Только `static/js`, `templates`, CSS | Достаточно деплоя файлов + поднять `?v=` + F5 у пользователя |
| `internal/*.py`, `.env`, `requirements.txt` | **Перезапуск** сервера / `docker compose up --build` |
| `layout.json` руками | Перезапуск или сохранение через API (вызовет `reload_layout`) |
| `TZ` в Docker | Пересборка и перезапуск контейнера |

Локально: `docker compose up --build` → http://localhost:5000  
VPS: `docker compose -f docker-compose.vps.yml up -d --build`

## 15. Типичные вопросы на защите — готовые ответы

**Почему монолит, а не микросервисы?**  
Учебный и демо-проект одного коворкинга; проще развёртывание, одна БД, одна кодовая база.

**Почему SVG, а не картинка плана?**  
Можно кликать по каждому месту, менять цвет статуса, масштабировать без потери качества.

**Почему JSON для планировки, а не всё в БД?**  
Геометрия (стены, координаты) удобно версионируется и быстро отдаётся карте; бизнес-данные — в PostgreSQL.

**Как избегаете двойного бронирования одного слота?**  
Проверка пересечений в `booking_service` перед записью; учёт `people_count` и `capacity`.

**Есть ли real-time?**  
Нет постоянного сокета; клиент периодически запрашивает timegrid и `/api/places` — для демо этого достаточно.

**Что если два человека одновременно бронируют одно место?**  
Оба запроса доходят до сервера; второй получит ошибку «время недоступно» после проверки в БД (транзакция).

**Как тестируете?**  
`py -3 -m pytest tests/ -v` — геометрия, коды мест.

**Чем менеджер отличается от админа?**  
Менеджер работает с клиентами и бронями; админ ещё правит планировку, тарифы, системные настройки.
