# Вопросы комиссии и обзор уязвимостей

## 30 вопросов по системе Smart Coworking

### Архитектура и назначение

1. **Что решает система?** Управление коворкингом: карта мест, бронирование по времени, тарифы, абонементы, роли (клиент, менеджер, админ).
2. **Почему веб-приложение, а не мобильное нативное?** Единый интерфейс для десктопа и телефона; карта и бронирование доступны через браузер без установки.
3. **Какой стек?** Python (Flask), PostgreSQL, SQLAlchemy, фронтенд — HTML/CSS/JS без тяжёлого SPA-фреймворка.
4. **Как развёртывается?** Docker + Waitress; конфигурация через `.env` (БД, секреты).
5. **Почему Flask, а не Django/FastAPI?** Лёгкий монолит с шаблонами Jinja2 — проще для учебного/демо-проекта с серверным рендерингом.

### Карта и SVG

6. **Почему SVG, а не Canvas или картинка?** Масштабирование без потери качества, клик по каждому месту, смена цвета статуса без перерисовки растра.
7. **Какой размер холста?** 2240×1344 px (масштаб 100 px = 1 м в редакторе).
8. **Где хранится планировка?** JSON-файл layout (`internal/layout/store.py`): места, стены, двери, координаты.
9. **Зачем дублировать в БД и в layout?** БД — бизнес-сущности (брони, статусы, тарифы); layout — геометрия и визуал. Связь по `code` места.
10. **Как связаны стены и комнаты?** Алгоритм `detect_all_wall_rooms` строит замкнутые контуры из сегментов стен; комната может быть зарегистрирована как `space` с `source=walls`.
11. **Почему столы — отдельные объекты, а не часть SVG?** Каждый стол — место с `kind=desk`, бронируется отдельно; геометрия обновляется при перетаскивании в редакторе.
12. **Как отображается занятость на карте?** API `/api/places` отдаёт статус (free/partial/occupied/maintenance); JS красит прямоугольники.
13. **Чем карта `/mapp` отличается от редактора?** `/mapp` — бронирование для пользователей; редактор — только для админов, с drag-and-drop и стенами.
14. **Как работает зум карты?** CSS `transform: scale()` на SVG, ограничение 1×–3×.

### Бронирование и время

15. **Какой шаг времени?** 15 минут; минимальная длительность — 30 минут.
16. **Откуда берётся расписание коворкинга?** Модель `Schedule` / `get_coworking_schedule_for_place` — часы работы по дням недели.
17. **Как считается занятость слота?** Сумма `people_count` активных броней на интервал; сравнение с `capacity` места.
18. **Почему время без timezone в API?** Бронирование — «настенные часы» коворкинга (Москва); клиент и сервер используют локальный/настроенный TZ (`Europe/Moscow`).
19. **Что такое timegrid?** Сетка 15-мин слотов с `occupied`, `available`, `status` для визуального выбора и валидации.
20. **Как тарифы влияют на форму?** Почасовой — дата+время; недельный/месячный — период без почасовой сетки.

### Геометрия и редактор

21. **Как предотвращается наложение столов?** `rects_overlap` с `gap=0`, `resolveDeskDragPosition` + скольжение вдоль осей и соседей (`slideRectOffDesks`).
22. **Можно ли ставить стол впритык к стене?** Да: `adjust_rect_from_walls` выталкивает прямоугольник к внутреннему краю полосы стены без пересечения.
23. **Проверка только на клиенте?** Нет — дублируется на сервере в `find_place_overlap` при сохранении (`/api/admin/place/move`).
24. **Что такое `effective_rect_for_rotation`?** Ограничивающий прямоугольник повёрнутого стола для коллизий.
25. **Зачем `container_code`?** Привязка стола к зоне/комнате для вместимости и отображения.

### Данные и безопасность

26. **Как хранятся пароли?** Хеш (Werkzeug); в БД не plaintext.
27. **Кто может двигать места на карте?** Только авторизованные админы (`@admin_required` на API layout).
28. **Есть ли CSRF?** `WTF_CSRF_ENABLED = True` для форм Flask-WTF.
29. **Как разграничены роли?** `client`, `manager`, `admin` — разные маршруты и проверки `is_manager()`, `is_admin()`.
30. **Что при ошибке геометрии при сохранении?** API возвращает 400 с текстом («Пересечение с…», «нельзя на стене»); клиент откатывает позицию.

## Выявленные риски и рекомендации

| Риск | Серьёзность | Описание | Рекомендация |
|------|-------------|----------|--------------|
| `SECRET_KEY` по умолчанию | Высокая | В `config.py` dev-ключ | Обязательный сильный ключ в production `.env` |
| `DEBUG = True` | Высокая | Утечка трассировок | `DEBUG=False` на проде |
| Доверие к клиентской геометрии | Средняя | Редактор валидирует на JS, но злоумышленник может слать API | Серверная проверка уже есть — не отключать |
| IDOR на бронирования | Средняя | Проверить, что клиент не может чужие `booking_id` | Аудит handlers booking + ownership |
| Нет rate limiting | Средняя | Brute-force логина / спам API | nginx limit_req или Flask-Limiter |
| Layout JSON на диске | Низкая | Race при параллельной записи | Уже есть atomic writes в store — проверить все пути |
| XSS в toast/innerHTML | Низкая | Редактор использует `escapeHtml` | Расширить на все пользовательские строки |
| UTC на сервере без TZ | Средняя | Исправлено: `Europe/Moscow` + клиентская дата | Контролировать `TZ` в Docker |
| Отсутствие HTTPS в dev | Инфо | Норма для локалки | TLS на проде |
| SQL injection | Низкая | SQLAlchemy ORM | Не использовать сырой SQL с конкатенацией |
