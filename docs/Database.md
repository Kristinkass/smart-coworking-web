# Разработка структуры базы данных

Перед реализацией базы данных для разрабатываемой системы управления коворкинг-пространством «Умный Коворкинг» был составлен перечень сущностей и их свойств. СУБД — **PostgreSQL**; доступ — **SQLAlchemy ORM**. Геометрия планировки (стены, двери, координаты мест) хранится в файле `static/layout.json`; бизнес-атрибуты мест — в таблице `places`.

## Условные обозначения в таблице

| Обозначение | Расшифровка |
|-------------|-------------|
| **PK** | Primary Key — первичный ключ (уникальный идентификатор записи) |
| **SERIAL** | Автоинкремент: значение назначается СУБД автоматически |
| **FK** | Foreign Key — внешний ключ (ссылка на запись в другой таблице) |
| **CASCADE** | При удалении родительской записи связанные записи удаляются автоматически |
| **SET NULL** | При удалении родительской записи внешний ключ обнуляется |
| **NOW()** | Текущие дата и время на момент создания записи |
| **NULL** | Значение отсутствует (поле необязательно или не заполнено) |
| **JSON** | Текст в формате JSON (структурированные данные в одном поле) |

## Справочник значений полей (расшифровка кодов)

### users.role — роль пользователя в системе

| Код | Значение |
|-----|----------|
| `client` | Клиент — самостоятельное бронирование, личный кабинет *(по умолчанию)* |
| `manager` | Менеджер (ресепшен) — управление бронированиями, клиентами, уведомлениями |
| `admin` | Администратор — полное управление системой, тарифами, планировкой, отчётами |

### users.visitor_kind — способ оплаты посещений (только для role = client)

| Код | Значение |
|-----|----------|
| `tariff` | Оплата по разовым тарифам (почасовой, недельный, месячный) *(по умолчанию)* |
| `subscription` | Оплата по оформленному абонементу (списание часов из лимита) |

### place_categories.kind / subscriptions.place_kinds — тип рабочего ресурса

| Код | Значение |
|-----|----------|
| `desk` | Рабочий стол (открытая зона или стол в комнате) |
| `room` | Переговорная комната |

### places.kind — тип объекта на интерактивной карте

| Код | Значение |
|-----|----------|
| `desk` | Стол — бронируется по местам за столом |
| `space` | Контейнер — закрытая зона столов или переговорная (бронируется целиком или по столам внутри) |
| `room` | Устаревший alias контейнера (эквивалент `space`, в новых данных не используется) |

**Важно:** в таблице `places` хранятся **только бронируемые объекты** — столы (`desk`) и переговорные/зоны столов (`space` с категорией `desk` или `room`). Служебные зоны (кухня, отдых, санузел) **не попадают** в `places`: для них достаточно записи в `locations` + геометрии в `layout.json`.

### places.status — отображаемый статус места

| Код | Значение |
|-----|----------|
| `free` | Свободно — доступно для бронирования *(по умолчанию)* |
| `occupied` | Занято — все места заняты на текущий интервал |
| `maintenance` | На обслуживании — бронирование недоступно |

*На карте также используется промежуточный статус `partial` (частично занято) — вычисляется динамически, в БД не хранится.*

### category_tariffs.tariff_type / bookings.tariff_type — тип тарифа

| Код | Значение |
|-----|----------|
| `hourly` | Почасовой тариф *(по умолчанию для брони)* |
| `weekly` | Недельный тариф |
| `monthly` | Месячный тариф |

### bookings.status — статус бронирования

| Код | Значение |
|-----|----------|
| `active` | Действует — бронь активна *(по умолчанию)* |
| `cancelled` | Отменена клиентом или менеджером |
| `completed` | Завершена — время аренды истекло |

### location_zone_types.kind / locations.kind — назначение зоны этажа

| Код | Значение | Бронирование | Запись в `places` |
|-----|----------|--------------|-------------------|
| `desk_zone` | Зона рабочих столов *(по умолчанию)* | Столы внутри зоны | Да — столы и закрытые контейнеры |
| `room_zone` | Зона переговорных комнат | Переговорные целиком | Да — контейнеры `space` с категорией `room` |
| `lounge_zone` | Зона отдыха (служебная) | Нет | **Нет** — только `locations` + `layout.json` |

## Типы связей между сущностями

### Иерархия пространства (этаж → зона → место)

```
coworking (1)
  └── floor (N этажей)
        └── location_zone_type (6 справочных типов: A/B/C/R/K/W)
              └── location (авто: по одной на каждый этаж × тип зоны)
                    └── place (только desk и room/space — бронируемые)
```

### location_zone_types → locations (автоматическое создание)

- **location_zone_types и locations**: один-ко-многим (1:N). Связь через `zone_type_id` в `locations`.
- **Система создаёт локации автоматически** при инициализации и при добавлении этажа: для каждого активного типа зоны (`LocationZoneType`) и каждого этажа (`Floor`) вызывается `ensure_location_for_zone(floor_num, zone_type_id)`.
- **Код локации** формируется как `{номер_этажа}{буква_зоны}`: этаж 3 + зона K → `3K`, этаж 1 + зона A → `1A`.
- **Поля `name` и `kind`** в `locations` копируются из типа зоны: например, для буквы K — «Кухня», `kind = kitchen_zone`.
- Локация — **логическая зона этажа**, а не конкретный прямоугольник на карте. Геометрия служебных зон (кухня, отдых) хранится в `layout.json`, но **без записи в `places`**.

| Тип (letter) | kind | Пример локаций | Что в `places` |
|--------------|------|----------------|----------------|
| A | desk_zone | 1A, 2A, 3A | Столы, закрытые зоны столов |
| B | room_zone | 1B, 2B, 3B | Переговорные |
| C | room_zone | 1C, 2C, 3C | Большие переговорные |
| R | lounge_zone | 1R, 2R, 3R | — (только layout) |
| K | kitchen_zone | 1K, 2K, 3K | — (только layout) |
| W | wc_zone | 1W, 2W, 3W | — (только layout) |

### locations → places

- **locations и places**: один-ко-многим (1:N). Каждое бронируемое место (`desk` или `space`) привязано к локации через `location_id`.
- Код места строится от локации: `1A-T108` (стол), `1B-L5` (переговорная), где префикс `1A` / `1B` — код локации.
- Служебные локации (`3R`, `3K`, `2K` …) **не имеют** дочерних записей в `places`. На карте их контур рисуется из `layout.json` по полю `location`.

### Прочие связи

- **coworkings и floors**: один-ко-многим (1:N). Связь реализована через внешний ключ `coworking_id` в таблице `floors`. При удалении коворкинга все его этажи удаляются автоматически (ON DELETE CASCADE).

- **coworkings и coworking_schedules**: один-ко-многим (1:N). Связь реализована через внешний ключ `id_coworking` в таблице `coworking_schedules`. Для каждого коворкинга хранится расписание работы по дням недели.

- **floors и locations**: один-ко-многим (1:N). Связь реализована через внешний ключ `floor_id` в таблице `locations`. На каждом этаже для каждого типа зоны создаётся одна локация с уникальным кодом (`1A`, `2B`, `3K` …). Локации создаются **автоматически** при `init_db()` — администратор задаёт только типы зон в справочнике `location_zone_types`.

- **floors и places**: один-ко-многим (1:N). Связь через `floor_id` (ON DELETE SET NULL). Дублирует этаж для быстрой выборки мест.

- **locations и places**: один-ко-многим (1:N). Связь через `location_id` (ON DELETE CASCADE). **Только бронируемые** объекты: столы и переговорные/контейнеры столов. Служебные зоны в `places` не хранятся.

- **place_categories и places**: один-ко-многим (1:N). Связь реализована через внешний ключ `category_id` в таблице `places` (ON DELETE SET NULL). Категория задаёт шаблон места: тип, вместимость, размеры.

- **place_categories и category_tariffs**: один-ко-многим (1:N). Связь реализована через внешний ключ `category_id` в таблице `category_tariffs` (ON DELETE CASCADE). Для каждой категории может быть несколько тарифов (часовой, недельный, месячный).

- **places и places (контейнер → стол)**: логическая связь один-ко-многим. Реализована через поле `container_code` в таблице `places` и в `layout.json`: для стола (`kind = desk`) в нём хранится код контейнера (`kind = space`). Это **не** внешний ключ: контейнер и стол — разные записи одной таблицы, связанные по строковому коду.

- **users и bookings**: один-ко-многим (1:N). Связь через `user_id` (**ON DELETE RESTRICT**). Пользователей не удаляют — только блокируют (`active = FALSE`), чтобы сохранить историю бронирований и отчёты по доходам.

- **places и bookings**: один-ко-многим (1:N). Связь через `place_id` (**ON DELETE RESTRICT**). Место с карты не удаляют из БД — ставят `active = FALSE` и убирают из `layout.json`; `code` и `name` остаются в таблице `places`, брони и отчёты по доходам сохраняются.

- **users и ratings**: один-ко-многим (1:N). Связь через `user_id` (**ON DELETE RESTRICT**).

- **places и ratings**: один-ко-многим (1:N). Связь через `place_id` (**ON DELETE SET NULL**).

- **category_tariffs и bookings**: один-ко-многим (1:N). Связь реализована через внешний ключ `category_tariff_id` в таблице `bookings` (ON DELETE SET NULL). Фиксирует тариф, по которому рассчитана стоимость брони.

- **subscriptions и bookings**: один-ко-многим (1:N). Связь реализована через внешний ключ `subscription_id` в таблице `bookings` (ON DELETE SET NULL). Если бронь оформлена по абонементу, сохраняется ссылка на него.

- **users и subscriptions**: один-ко-многим (1:N). Связь реализована через внешний ключ `user_id` в таблице `subscriptions` (ON DELETE CASCADE). Поле `user_id` может быть NULL для шаблонов абонементов, доступных к покупке.

- **bookings и ratings**: один-ко-многим (1:N). Связь реализована через внешний ключ `booking_id` в таблице `ratings` (ON DELETE CASCADE). Позволяет привязать оценку к конкретному визиту.

- **users и notifications**: один-ко-многим (1:N). Связь реализована через внешний ключ `user_id` в таблице `notifications` (ON DELETE CASCADE). Используется для персональных уведомлений; при массовой рассылке поле может быть NULL.

- **users и notifications (отправитель)**: один-ко-многим (1:N). Связь через `sender_id` (ON DELETE SET NULL). Фиксирует администратора, менеджера или клиента (обратная связь по брони).

- **bookings и notifications**: один-ко-многим (1:N). Связь через `booking_id` (ON DELETE SET NULL). Используется для сообщений клиента менеджеру/админу с привязкой к конкретной брони.

- **layout.json и places**: логическая связь один-к-одному (1:1) по полю `code`. Геометрия (координаты, размеры, стены) хранится в JSON-файле; бизнес-атрибуты (статус, рейтинг, тарифная категория) — в PostgreSQL.

## IDEF1X-диаграмма

Для представления структуры базы данных разработана диаграмма в нотации **IDEF1X** (Integrated Definition for Information Modeling). В отличие от нотации Crow's Foot, IDEF1X явно разделяет первичные ключи и прочие атрибуты горизонтальной чертой, а нетождественные (non-identifying) связи изображаются **пунктирными линиями** — все связи в данной схеме относятся к этому типу, поскольку первичные ключи сущностей — суррогатные идентификаторы (`SERIAL`), а внешние ключи не входят в состав PK.

Диаграмма представлена на рисунке (файл `docs/ER-diagram.drawio`).

**Рисунок — IDEF1X-диаграмма базы данных системы «Умный Коворкинг»**

Условные обозначения:
- подчёркнутый атрибут — первичный ключ;
- атрибут с `*` — внешний ключ с необязательным участием (допускает NULL);
- пунктирная линия — нетождественная связь «один ко многим».

Проектирование базы данных выполнено с разделением данных по функциональному назначению. Информация о пользователях, этажах и зонах, рабочих местах, бронированиях, тарифах, абонементах, расписании, уведомлениях и оценках хранится в отдельных таблицах, что упрощает сопровождение данных и обеспечивает их логическую целостность. Геометрия планировки вынесена в отдельный файл `layout.json`, что позволяет редактировать карту этажа без изменения реляционной схемы.

Для хранения данных в системе «Умный Коворкинг» была выбрана система управления базами данных **PostgreSQL**. Данное решение обусловлено тем, что разрабатываемый программный продукт представляет собой веб-приложение с одновременной работой клиентов, менеджеров и администраторов, а также требует надёжного хранения транзакционных данных о бронированиях.

Одним из основных преимуществ PostgreSQL является полноценная поддержка ACID-транзакций, внешних ключей и каскадного удаления, что критично при создании и отмене бронирований, списании часов абонемента и обновлении статусов мест. СУБД обеспечивает целостность данных при параллельных запросах нескольких пользователей к одним и тем же рабочим местам.

Кроме того, PostgreSQL отличается высокой производительностью при работе с реляционными связями, поддерживает индексы по уникальным полям (`email`, `code`) и масштабируется при росте числа бронирований и пользователей. Использование PostgreSQL в связке с SQLAlchemy ORM позволяет описывать сущности на уровне объектной модели Python и автоматически синхронизировать их со схемой базы данных.

Таким образом, выбор PostgreSQL обусловлен его надёжностью, поддержкой транзакций и соответствием требованиям многопользовательской информационной системы управления коворкингом.

## Дополнительное хранилище: layout.json

Файл планировки содержит **только геометрию** этажа. Справочники зон и тарифов в нём не хранятся.

| Раздел JSON | Назначение |
|-------------|------------|
| `coworking` | Метаданные площадки (имя, адрес) |
| `floors[]` | Этажи: номер, размер холста |
| `grid` | Сетка редактора (cols, rows, cell) |
| `walls[]` | `id`, `x1`, `y1`, `x2`, `y2`, `floor` |
| `doors[]` | `id`, `wall_id`, `position`, `width`, `floor` |
| `places[]` | `code`, `x`, `y`, `width`, `height`, `floor`, `kind`, `location`, `container_code`, `enclosed`, `category_id`, `zone_type_id`, `bookable` |

**Служебные зоны** (кухня, отдых): геометрия в `places[]` с `bookable: false` и `location: "3K"` и т.п., но **без строки в таблице PostgreSQL `places`**. API карты отдаёт их из layout, подставляя `name`/`kind` из связанной `locations`.

Синхронизация с БД: `sync_place_by_code()`, `layout_place_belongs_in_db()`, `purge_amenity_places()` (`internal/models/seed.py`, `internal/models/location_zone.py`).
