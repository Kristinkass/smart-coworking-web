# 2.4. Разработка алгоритмов функционирования программного обеспечения

## Как алгоритмы отличаются от BPMN (§2.2.5)

В разделе 2.2.5 описаны **бизнес-процессы** в нотации BPMN: кто выполняет действие (клиент, менеджер, администратор), в каком порядке и при каких условиях процесс переходит к следующему шагу. Например, процесс «Создание бронирования» включает шаги «выбрать место на карте», «нажать „Забронировать“», «система проверяет доступность», «система сохраняет бронь».

Раздел 2.4 описывает **алгоритмы** — правила вычислений и автоматических проверок, которые система выполняет **внутри** отдельных шагов BPMN. Здесь нет дорожек участников: только входные данные, формулы, условия, циклы и результат.

| BPMN уже описывает (§2.2.5) | Алгоритмы раскрывают (§2.4) |
|-----------------------------|-----------------------------|
| Клиент выбирает место и время | Как разбить день на слоты и посчитать занятость \(s_i\) |
| «Система проверяет доступность» | Формулы overlap, \(s_i + p \leq C\), ограничения длительности |
| «Система рассчитывает стоимость» | Формулы для hourly / weekly / monthly и абонемента |
| Карта обновляется после брони | Как по \(s_i\) и \(C\) определить цвет места |
| Администратор размещает место в редакторе | Как перевести метры в пиксели и проверить пересечение |

**Не дублируем в §2.4:** вход в систему, регистрация, отмена/продление брони, техобслуживание, оценка места, отчёт, абонемент, рассылка уведомлений — для них достаточно BPMN и DFD.

**В §2.4 включаем 4 алгоритма** — вычислительное ядро системы, уникальное для «Умного Коворкинга».

---

## Вводный абзац (для пояснительной записки)

Для обеспечения корректной работы веб-приложения «Умный Коворкинг» разработаны алгоритмы функционирования ключевых программных модулей. В отличие от BPMN-диаграмм (раздел 2.2.5), описывающих последовательность действий участников, алгоритмы настоящего раздела определяют **математические правила и логику автоматических операций**: дискретизацию рабочего дня, расчёт занятости мест, проверку допустимости бронирования, тарификацию и формирование статусов на интерактивной карте. Ниже представлены четыре основных алгоритма, реализованные в серверной части приложения (Python, PostgreSQL).

## 2.4.1. Алгоритм расчёта занятости по временным слотам

### Назначение

Алгоритм реализует **внутреннюю логику** шага «проверить доступность» из BPMN-процесса создания бронирования. Рабочий день коворкинга разбивается на отрезки по 15 минут; для каждого слота вычисляется, сколько человек уже забронировало выбранное место. Это позволяет бронировать **часть многоместного стола** — например, два человека с 10:00 до 12:00 и ещё два с 12:00 до 14:00 за одним четырёхместным столом.

### Входные данные

- расписание коворкинга: \(T_{open}\), \(T_{close}\) — время открытия и закрытия;
- множество активных броней места на выбранную дату;
- запрашиваемый интервал \([T_{start}, T_{end}]\);
- \(p\) — число человек в новой брони;
- \(C\) — вместимость места.

### Формулы

**Число слотов рабочего дня:**

\[
N_{slots} = \left\lfloor \frac{T_{close} - T_{open}}{\Delta t} \right\rfloor \tag{4}
\]

где \(\Delta t = 15\) мин.

**Индекс слота для времени \(t\):**

\[
i(t) = \left\lfloor \frac{t - T_{open}}{\Delta t} \right\rfloor \tag{5}
\]

**Занятость слота \(i\):**

\[
s_i = \sum_{b \in B_i} p_b \tag{6}
\]

где \(B_i\) — множество броней, пересекающих слот \(i\); \(p_b\) — поле `people_count` брони \(b\).

**Условие допустимости по вместимости в слоте:**

\[
s_i + p \leq C \tag{7}
\]

**Пересечение интервала брони \([t_1^{(b)}, t_2^{(b)})\) и слота \([t_1^{(s)}, t_2^{(s)})\)** (бронь \(b\) входит в \(B_i\), если):

\[
\text{overlap}(b,s) = \bigl(t_1^{(b)} \leq t_1^{(s)} < t_2^{(b)}\bigr) \lor \bigl(t_1^{(b)} < t_2^{(s)} \leq t_2^{(b)}\bigr) \lor \bigl(t_1^{(s)} \leq t_1^{(b)} \land t_2^{(s)} \geq t_2^{(b)}\bigr) \tag{8}
\]

### Этапы

1. Получить \(T_{open}\), \(T_{close}\) из таблицы `coworking_schedules`.
2. Вычислить индексы слотов \(i_{start}\), \(i_{end}\) для \([T_{start}, T_{end}]\) по формуле (5).
3. Для каждого \(i\) от \(i_{start}\) до \(i_{end} - 1\):
   - сформировать \(B_i\) по условию (8);
   - вычислить \(s_i\) по формуле (6);
   - проверить (7); при нарушении — **отказ**.
4. Если все слоты прошли проверку — **допустимо**.

### Правило длительности (для интерфейса)

Начало брони — с шагом 15 мин; **длительность** — только 30, 60, 90 мин и т.д. (чётное число слотов):

\[
\frac{T_{end} - T_{start}}{\Delta t} = 2n,\quad n \geq 1 \tag{8а}
\]

## 2.4.2. Алгоритм проверки допустимости бронирования

### Назначение

Обобщает результат алгоритма 2.4.1 и добавляет **бизнес-ограничения**, не входящие в формулы занятости: расписание, прошедшее время, режим техобслуживания, правило «переговорная только целиком», конфликт «стол vs помещение целиком». В BPMN это один блок «проверить доступность»; здесь — полная логика решения «да/нет».

### Входные данные

Идентификатор места, дата, \(T_{start}\), \(T_{end}\), \(p\), тип тарифа.

### Дополнительные условия (без отдельных формул)

| № | Условие | При нарушении |
|---|---------|---------------|
| 1 | Коворкинг работает в выбранный день | Отказ |
| 2 | \([T_{start}, T_{end}] \subseteq [T_{open}, T_{close}]\) | Отказ |
| 3 | Дата и время не в прошлом | Отказ |
| 4 | \(T_{start}\), \(T_{end}\) кратны 15 мин; длительность по (8а) | Отказ |
| 5 | Место не в режиме `maintenance` | Отказ |
| 6 | Алгоритм 2.4.1: \(s_i + p \leq C\) во всех слотах | Отказ |
| 7 | Переговорная: \(p = C\) | Отказ, если \(p \neq C\) |
| 8 | Нет конфликта «помещение целиком ↔ отдельный стол» | Отказ |

Для **недельного** (\(P = 7\)) и **месячного** (\(P = 30\)) тарифов множество дат:

\[
D = \{d_0,\, d_0 + 1,\, \ldots,\, d_0 + P - 1\} \tag{9}
\]

На каждый рабочий день \(d \in D\) выполняется проверка занятости на весь рабочий интервал.

Схема алгоритма проверки допустимости бронирования представлена на **рисунке 21**.

## 2.4.5. Алгоритм расчёта стоимости бронирования

### Назначение

Алгоритм выполняется после успешной проверки доступности (§2.4.4) — на шаге BPMN «рассчитать стоимость». Определяет итоговую сумму бронирования, сохраняемую в базе данных, и сумму, отображаемую клиенту в форме бронирования. Реализация — модуль `internal/handlers/booking/api.py`.

### Условные обозначения (общие для формул §2.4.5)

| Обозначение | Что означает |
|-------------|--------------|
| \(T_{start}\), \(T_{end}\) | Время начала и окончания брони (часы и минуты), кратно 15 мин |
| \(t_{start}\), \(t_{end}\) | То же время в **минутах от полуночи** (8:00 → 480, 10:30 → 630) |
| \(\Delta t\) | Длительность одного временного слота; в системе \(\Delta t = 15\) мин |
| \(N_{slots}\) | Число 15-минутных слотов в выбранном интервале |
| \(\Delta h\) | Длительность брони в **часах** (дробное, кратно 0,25) |
| \(P_{hour}\) | Цена **почасового** тарифа для категории места, руб./ч |
| \(P_{tariff}\) | Фиксированная цена **недельного** или **месячного** тарифа, руб. |
| \(p\) | Число человек в новой брони (`people_count`), \(p \geq 1\) |
| \(C\) | Вместимость места (число мест за столом или в зоне) |
| \(P_{calc}\) | Рассчитанная стоимость **до** учёта абонемента, руб. |
| \(P_{total}\) | Итоговая сумма **к оплате** (с учётом абонемента), руб. |
| \(d_0\) | Дата начала бронирования |
| \(H_{limit}\) | Лимит часов по абонементу (если задан) |
| \(H_{used}\) | Уже использовано часов по абонементу |
| \(\Phi\) | Признак «цена за всё помещение»: 1 — переговорная или зона целиком, 0 — отдельный стол |

Признак \(\Phi\) определяется типом объекта: для переговорной комнаты или зоны столов, бронируемой целиком, тариф задаётся на **всё помещение**; для рабочего стола — **на одного человека**, поэтому сумма умножается на \(p\).

---

### Формула (12). Длительность почасового бронирования

Сначала время переводится в минуты от полуночи:

\[
t_{start} = H_{start} \cdot 60 + M_{start}, \qquad
t_{end} = H_{end} \cdot 60 + M_{end} \tag{12а}
\]

где \(H_{start}, M_{start}\) — часы и минуты начала; \(H_{end}, M_{end}\) — часы и минуты окончания.

Число слотов и длительность в часах:

\[
N_{slots} = \left\lfloor \frac{t_{end} - t_{start}}{\Delta t} \right\rfloor, \qquad
\Delta h = N_{slots} \cdot 0{,}25 \tag{12}
\]

**Пояснение:** один слот = 15 мин = 0,25 ч. Округление вниз (\(\lfloor \cdot \rfloor\)) гарантирует согласованность с алгоритмом занятости (§2.4.3): бронь и проверка доступности используют одну и ту же сетку.

**Пример:** 10:00–11:30 → \(t_{start}=600\), \(t_{end}=690\) → \(N_{slots} = \lfloor 90/15 \rfloor = 6\) → \(\Delta h = 6 \cdot 0{,}25 = 1{,}5\) ч.

---

### Формула (13). Почасовой тариф — рабочий стол (\(\Phi = 0\))

\[
P_{calc} = P_{hour} \cdot \Delta h \cdot p \tag{13}
\]

| Сомножитель | Смысл |
|-------------|--------|
| \(P_{hour}\) | Стоимость одного часа аренды **одного места** за столом |
| \(\Delta h\) | Сколько часов длится бронь |
| \(p\) | Сколько человек бронирует места за этим столом |

**Пример:** \(P_{hour} = 200\) руб., \(\Delta h = 2\) ч, \(p = 3\) чел. → \(P_{calc} = 200 \cdot 2 \cdot 3 = 1200\) руб.

---

### Формула (14). Почасовой тариф — переговорная или зона целиком (\(\Phi = 1\))

\[
P_{calc} = P_{hour} \cdot \Delta h \tag{14}
\]

**Пояснение:** цена не умножается на \(p\), так как клиент арендует **всё помещение** (переговорную или закрытую зону столов), а не отдельное место.

**Пример:** \(P_{hour} = 800\) руб., \(\Delta h = 1{,}5\) ч → \(P_{calc} = 1200\) руб.

---

### Формула (15). Недельный или месячный тариф — рабочий стол (\(\Phi = 0\))

\[
P_{calc} = P_{tariff} \cdot p \tag{15}
\]

| Сомножитель | Смысл |
|-------------|--------|
| \(P_{tariff}\) | Фиксированная цена тарифа за **весь период** (7 или 30 дней) для одного человека |
| \(p\) | Число человек, на которых оформляется бронь |

**Пояснение:** для периодного тарифа время суток внутри дня не влияет на цену — оплачивается весь период \(D\) из формулы (11) §2.4.4.

---

### Формула (16). Недельный или месячный тариф — переговорная или зона (\(\Phi = 1\))

\[
P_{calc} = P_{tariff} \tag{16}
\]

**Пояснение:** фиксированная цена за помещение на весь оплаченный период, без умножения на число людей.

---

### Формула (17). Округление до рублей

\[
P_{calc} = \mathrm{round}(P_{calc}) \tag{17}
\]

**Пояснение:** в системе сумма хранится и отображается **целым числом рублей** (без копеек).

---

### Формула (18). Учёт абонемента

Абонемент применяется только к **почасовому** тарифу. Условие допустимости:

\[
A = (d_0 \in [S_{sub}, E_{sub}]) \land K_{place} \land (H_{limit} = \varnothing \lor H_{used} + \Delta h \leq H_{limit}) \tag{18а}
\]

где:
- \(S_{sub}, E_{sub}\) — даты начала и окончания действия абонемента;
- \(K_{place}\) — тип места входит в перечень, разрешённый абонементом (стол, переговорная, зона);
- \(H_{limit} = \varnothing\) — безлимитный абонемент;
- \(H_{used}\) — уже израсходовано часов по абонементу.

Итоговая сумма к оплате:

\[
P_{total} =
\begin{cases}
0, & \text{если } A = \text{истина} \\
P_{calc}, & \text{иначе}
\end{cases} \tag{18}
\]

**Пояснение:** при \(A = \text{истина}\) клиент не платит за бронь, но полная рассчитанная сумма \(P_{calc}\) сохраняется для отображения («стоимость без абонемента»). После сохранения брони: \(H_{used} := H_{used} + \Delta h\).

---

### Этапы алгоритма

1. Загрузить место и тариф категории по идентификатору места и типу тарифа.
2. Проверить: для закрытой зоны столов бронь оформляется только целиком.
3. Определить \(\Phi\) (стол — 0, переговорная/зона — 1).
4. Если тариф почасовой — вычислить \(\Delta h\) по (12).
5. Вычислить \(P_{calc}\) по (13)–(16), округлить по (17).
6. Если почасовой тариф — проверить абонемент по (18а); при успехе \(P_{total} = 0\).
7. Вернуть \(P_{total}\) и \(P_{calc}\) для отображения в форме; при сохранении брони записать \(P_{total}\) в базу данных.

Схема алгоритма расчёта стоимости бронирования представлена на **рисунке 22**.

## 2.4.7. Алгоритм синхронизации планировки с базой данных

### Назначение

Обеспечивает согласованность геометрии (файл `layout.json`) и бизнес-атрибутов (PostgreSQL). Выполняется при сохранении редактора планировки и при инициализации системы.

### Суть

Для каждого объекта на карте существует уникальный **код** `code`. Связь «один к одному»:

\[
\text{layout.json}[code] \leftrightarrow \text{places}[code]
\]

При сохранении редактора:
1. обновляются координаты, размеры, тип (`desk`, `room`, `space`) в JSON;
2. модуль синхронизации создаёт или обновляет запись в таблице `places` с тем же `code`;
3. переносятся `container_code` (стол внутри зоны), `enclosed`, этаж, категория.

**Пояснение:** разделение нужно, чтобы карта загружалась быстро (JSON), а транзакции бронирования шли через СУБД с гарантией целостности.

## 2.4.8. Алгоритм определения комнаты по точке на карте

### Назначение

Используется в редакторе планировки: администратор кликает внутри контура стен, система определяет прямоугольник комнаты для размещения переговорной или зоны столов.

### Формулы

Пусть \((x, y)\) — координаты клика на SVG-холсте. Находятся ближайшие стены:

\[
x_{left} = \max\{ x_w \mid x_w < x \}, \quad
x_{right} = \min\{ x_w \mid x_w > x \}
\]

\[
y_{top} = \max\{ y_w \mid y_w < y \}, \quad
y_{bottom} = \min\{ y_w \mid y_w > y \}
\]

Прямоугольник комнаты:

\[
R = (x_{left},\; y_{top},\; x_{right} - x_{left},\; y_{bottom} - y_{top}) \tag{19}
\]

**Условие валидности:** ширина и высота \(R\) не менее 80 px; стены образуют замкнутый прямоугольник (допуск по стыку \(\varepsilon = 8\) px).

## 2.4.9. Алгоритм поиска свободных интервалов заданной длительности

### Назначение

Подбирает все интервалы времени, когда место можно забронировать на \(L\) минут при \(p\) человеках (подсказки в форме бронирования).

### Формулы

Пусть построена последовательность слотов \(i = 0, 1, \ldots, N_{slots}-1\) с занятостью \(s_i\) и вместимостью \(C\). Требуется окно из \(K = L / \Delta t\) подряд идущих слотов:

\[
\forall j \in \{i, i+1, \ldots, i+K-1\}:\quad s_j + p \leq C \tag{20}
\]

Если условие (20) выполнено для начального индекса \(i\), интервал \([t_i, t_{i+K})\) **доступен**. Перебор всех \(i\) от \(0\) до \(N_{slots} - K\) даёт список свободных окон.

**Пример:** \(L = 30\) мин → \(K = 2\) слота; при \(C = 4\), \(p = 2\) нужно, чтобы в обоих слотах было не более \(4 - 2 = 2\) занятых мест.

## 2.4.10. Алгоритм расчёта агрегированной статистики для отчётов

### Назначение

Формирует показатели на страницах «Статистика» и «Отчёты»: доход, число броней, загрузка по дням и категориям.

### Формулы

**Доход за период** \([d_1, d_2]\):

\[
R = \sum_{b \in B} P_b \tag{21}
\]

где \(B\) — множество броней со статусом «активна» или «завершена», дата которых попадает в период; \(P_b\) — поле `total_price` брони \(b\).

**Загрузка места за день** (доля занятых слотов):

\[
U = \frac{\sum_{i=0}^{N_{slots}-1} \mathbb{1}[s_i > 0]}{N_{slots}} \cdot 100\% \tag{22}
\]

где \(\mathbb{1}[\cdot]\) — индикаторная функция (1, если слот занят хотя бы одним человеком).

**Средний рейтинг места** по \(n\) оценкам \(r_1, \ldots, r_n\):

\[
\bar{r} = \frac{1}{n} \sum_{k=1}^{n} r_k \tag{23}
\]

## 2.4.6. Алгоритм определения статуса места на интерактивной карте

### Назначение

Реализует логику цветовой индикации на SVG-карте (шаг BPMN «обновить карту»). Определяет текстовый статус места (`free`, `partial`, `occupied`, `maintenance`) и соответствующий цвет прямоугольника на карте. Реализация: функции `live_status`, `display_status` в `internal/handlers/map/places.py`, а также `compute_live_status`, `compute_container_status` в модели `Place`.

### Входные данные

- объекты из `layout.json` (координаты, `code`, `kind`, этаж);
- записи `places` из PostgreSQL (вместимость, `maintenance`, категория);
- активные брони (`status = 'active'`), пересекающие **текущий момент** (для общей карты) или выбранный интервал (для временной шкалы слотов в форме бронирования).

### Расшифровка функций

**`live_status(db_place)`** — базовый расчёт для одного места на **текущий момент**:
- если `db_place.maintenance == True` → статус `maintenance` (серый, бронь недоступна);
- иначе загружаются активные брони места, пересекающие текущее время;
- `current_occupancy = Σ people_count` по этим броням;
- `capacity = place.capacity` (или 1, если не задано);
- если `current_occupancy == 0` → `free` (зелёный);
- если `0 < current_occupancy < capacity` при `capacity > 1` → `partial` (жёлтый);
- иначе → `occupied` (красный).

Дополнительно формируются: `occupied_until` (время окончания последней брони), `partial_occupancy` (занято / вместимость / свободно), `whole_table_taken` (есть ли бронь на всю вместимость).

**`display_status(db_place)`** — статус для **контейнеров** (зон с дочерними столами). Если у зоны есть дочерние столы и на самой зоне нет прямой брони, занятость **агрегируется** по всем столам внутри: `total_occ = Σ occupancy(стол)`, `total_cap = Σ capacity(стол)`. По суммарным значениям применяются те же правила free / partial / occupied.

**`compute_live_status()` / `compute_container_status()`** — серверная версия той же логики в модели `Place`; используется при формировании `place.to_dict()` и обновлении поля `status` в БД.

**API `/api/places`** — объединяет геометрию из `layout.json` и статус из `display_status`, возвращает JSON-массив `places` для отрисовки SVG. Служебные зоны (`is_amenity`: кухня, санузел и т.п.) помечаются `bookable: false` и не участвуют в бронировании.

**API `/api/booking/timegrid/<place_id>`** — для выбранной **даты** строит 15-минутные слоты (алгоритм 2.4.3); каждый слот получает статус `free` / `partial` / `full` по занятости `occupied_seats` и `capacity`. Клиентская часть (`booking.js`) окрашивает слоты временной шкалы по тем же правилам, что и карта.

### Соответствие статуса и цвета

| Условие | Код статуса | Цвет на карте | Смысл |
|---------|-------------|---------------|-------|
| `maintenance = true` | `maintenance` | Серый | Техобслуживание, бронь запрещена |
| `current_occupancy = 0` | `free` | Зелёный | Свободно |
| `0 < current_occupancy < C` | `partial` | Жёлтый | Частично занято (многоместный стол) |
| `current_occupancy ≥ C` | `occupied` | Красный | Занято полностью |
| служебная зона | — | Без брони | Не кликабельно для бронирования |

Для интервала бронирования на временной шкале вместо `current_occupancy` используется `occupied_seats` в каждом слоте; при выборе пользователем `[T_start, T_end]` итоговый цвет места определяется по **максимальной** занятости среди пересекающихся слотов (аналогично проверке в алгоритме 2.4.4).

### Этапы алгоритма

1. Загрузить `layout.json` и список мест из PostgreSQL; сопоставить записи по полю `code`.
2. Выбрать все активные брони на текущую дату, пересекающие текущее время (или выбранный интервал).
3. Для каждого объекта на карте вызвать `display_status`.
4. Присвоить цвет по таблице соответствия.
5. Сформировать JSON (`status`, `current_occupancy`, `partial_occupancy`, координаты, тарифы) и передать клиенту.
6. JavaScript-модуль `map_updated.js` отрисовывает SVG-прямоугольники с классом цвета по полю `status`.
