# 3. ПРОГРАММНАЯ РЕАЛИЗАЦИЯ СИСТЕМЫ «УМНЫЙ КОВОРКИНГ»

## 3.1 Выбор средств

Для программной реализации системы управления рабочими местами в коворкинге был осуществлён выбор инструментов разработки, управления данными и тестирования. При выборе средств учитывались следующие критерии: функциональная полнота, удобство работы в команде и в индивидуальном режиме, бесплатное или открытое распространение, кроссплатформенная доступность и достаточность для реализации всех заявленных функций системы.

Разработка серверной части выполнена на языке программирования **Python** версии 3.10 и выше [1]. Данный выбор обусловлен тем, что Python занимает лидирующие позиции в области веб-разработки и анализа данных, обладает обширной экосистемой библиотек для работы с реляционными базами данных, HTTP-запросами и формированием документов. По сравнению с языком Java или платформой Node.js, Python предоставляет более лаконичный синтаксис и меньший объём шаблонного кода при построении REST API и серверных шаблонов, что существенно ускоряет разработку в рамках дипломного проекта. В отличие от Go или Rust, Python обладает значительно более низким порогом входа, а его стандартная библиотека покрывает большую часть прикладных задач — работу с JSON, файловой системой, HTTP и криптографией паролей — без привлечения дополнительных зависимостей.

В качестве основной **среды разработки** была выбрана **PyCharm** компании JetBrains [2]. Данный выбор обусловлен рядом преимуществ перед альтернативными решениями, такими как Visual Studio Code или Sublime Text. PyCharm изначально ориентирован на экосистему Python и обеспечивает глубокую статическую проверку типов, навигацию по определениям символов ORM-моделей SQLAlchemy и автодополнение кода в шаблонах Jinja2. Visual Studio Code, несмотря на широкое распространение и богатую библиотеку расширений, требует ручной настройки виртуального окружения, линтеров и плагинов для каждого проекта, тогда как PyCharm распознаёт структуру проекта и файл `requirements.txt` автоматически. Sublime Text является лёгким текстовым редактором и не обеспечивает интеграции с системами контроля версий и инструментами отладки без значительной дополнительной настройки. PyCharm, напротив, предоставляет встроенный отладчик с пошаговым исполнением, профилировщик, интеграцию с Git и панель управления базами данных. Это позволило проводить отладку логики бронирования и синхронизации планировки непосредственно в среде разработки без переключения между сторонними инструментами.

Структура проекта в PyCharm организована в соответствии с принятой многослойной архитектурой: каталог `cmd/` содержит точку входа приложения, `internal/` — прикладную бизнес-логику, `templates/` и `static/` — клиентские ресурсы, `tests/` — автоматизированные тесты.

Для реализации **серверной части** применён микрофреймворк **Flask** версии 2.3.x [3]. В сравнении с полнофункциональным фреймворком Django, Flask не навязывает конкретную структуру проекта и не включает встроенных компонентов для административного интерфейса, форм и ORM, которые не нужны в данной системе или заменены специализированными решениями. Django избыточен для проекта, где требуется гибкое разделение на уровни handlers, services и repositories, соответствующее предметной области коворкинга. FastAPI ориентирован преимущественно на асинхронные высоконагруженные API и требует дополнительных усилий при интеграции с серверными HTML-шаблонами. Flask, напротив, органично объединяет серверный рендеринг страниц через движок Jinja2 и REST API для интерактивной карты, что полностью соответствует архитектуре разрабатываемой системы [3, 4]. Механизм Blueprint позволяет логически сгруппировать маршруты бронирования, карты и администрирования, а расширение Flask-Login обеспечивает управление сессией пользователя и ролевой моделью [5].

Для **хранения данных** выбрана реляционная СУБД **PostgreSQL** [7]. В сравнении с MySQL или MariaDB, PostgreSQL обеспечивает более строгую поддержку ACID-транзакций и богатый набор типов данных, что критически важно при конкурентном бронировании одного рабочего места несколькими клиентами. В сравнении со встраиваемой СУБД SQLite, PostgreSQL рассчитана на параллельный доступ множества клиентов и поддерживает декларативные внешние ключи с политиками каскадного обновления и ограничения удаления, что обеспечивает сохранность финансовой истории бронирований при снятии места с карты. Подключение к серверу PostgreSQL выполняется через драйвер psycopg2, а взаимодействие с базой данных на уровне приложения — через ORM SQLAlchemy в составе расширения Flask-SQLAlchemy [9, 10].

Для **управления базой данных** и визуального представления схемы таблиц использовался **pgAdmin 4** — официальный графический клиент PostgreSQL [11]. По сравнению с универсальным клиентом DBeaver или командной строкой `psql`, pgAdmin 4 глубоко интегрирован с экосистемой PostgreSQL, предоставляет наглядное дерево объектов, Query Tool для ad-hoc запросов и инструменты резервного копирования. Это позволяло в ходе разработки визуально проверять корректность внешних ключей, соответствие физической схемы ER-диаграмме из главы 2 и состояние данных после тестовых сценариев бронирования.

## 3.2 Реализация хранения данных

Для хранения бизнес-данных системы используется реляционная СУБД PostgreSQL. Доступ к данным из приложения организован через ORM SQLAlchemy: единственный экземпляр `db` объявлен в модуле `internal/models/db.py`, а все модели сущностей размещены в пакете `internal/models/`. База данных содержит семь основных таблиц, соответствующих ключевым сущностям предметной области.

Таблица **users** предназначена для хранения учётных записей пользователей системы. Таблица содержит следующие поля: `id_user` — целочисленный первичный ключ с автоинкрементом; `email` — адрес электронной почты, уникальный для каждого пользователя, является основным идентификатором при входе; `username` — отображаемое имя; `password_hash` — хеш пароля, вычисляемый средствами Werkzeug; `role` — текстовое поле, определяющее права доступа пользователя (значения `admin`, `manager`, `client`); `phone` — необязательный номер телефона; `visitor_kind` — способ расчёта клиента (`tariff` или `subscription`); `active` — булев флаг, при установке в `FALSE` блокирует вход без физического удаления записи; `last_login` — дата и время последнего успешного входа. Пользователи не удаляются из базы данных физически — при необходимости ограничения доступа устанавливается `active = FALSE`, что позволяет сохранять всю историю бронирований, связанных с данной учётной записью.

Код модели таблицы users представлен на рисунке 32.

```python
class User(UserMixin, db.Model):
    __tablename__ = 'users'
    id_user      = db.Column(db.Integer, primary_key=True)
    email        = db.Column(db.String(120), unique=True, nullable=False)
    role         = db.Column(db.String(20), default='client')
    active       = db.Column(db.Boolean, default=True)
    visitor_kind = db.Column(db.String(20), default='tariff')
```

Таблица **places** хранит бизнес-атрибуты рабочих мест коворкинга. Поля таблицы: `id_place` — первичный ключ; `code` — уникальный текстовый код места, используемый для связи с геометрией в файле `static/layout.json`; `name` — отображаемое название; `category_id` — внешний ключ на таблицу категорий; `location_id` — внешний ключ на локацию (зону этажа); `status` — текущий статус места (`free`, `occupied`, `maintenance`); `kind` — тип объекта (`desk`, `space`, `room`); `enclosed` — булев флаг, обозначающий закрытое помещение. Таблица **bookings** фиксирует все бронирования рабочих мест. Поля таблицы: `id_booking` — первичный ключ; `user_id` — внешний ключ на пользователя с политикой `ON DELETE RESTRICT`, исключающей физическое удаление пользователя при наличии его бронирований; `place_id` — внешний ключ на место с той же политикой `RESTRICT`, благодаря чему история бронирований сохраняется даже при снятии рабочего места с интерактивной карты; `booking_date` — дата начала брони; `start_time` и `end_time` — границы забронированного временного интервала; `tariff_type` — тип тарифа (`hourly`, `weekly`, `monthly`); `total_price` — итоговая стоимость в рублях; `people_count` — количество посетителей; `status` — статус брони (`active`, `cancelled`, `completed`). Таблица **subscriptions** хранит как шаблоны абонементных планов, так и активные абонементы клиентов. Поля: `id_subscription` — первичный ключ; `is_template` — булев флаг, отделяющий шаблоны от персональных абонементов; `user_id` — внешний ключ на пользователя (для персональных абонементов); `name` — название плана; `duration_days` — срок действия в днях; `hours_limit` — максимальное количество часов по абонементу; `price` — стоимость; `place_kinds` — JSON-строка с перечнем типов мест, доступных по данному абонементу; `start_date` и `end_date` — даты начала и окончания действия; `remaining_hours` — остаток доступных часов.

Таблица **coworking_schedules** задаёт режим работы коворкинга по дням недели. Поля: `id_schedule` — первичный ключ; `coworking_id` — внешний ключ на запись о коворкинг-пространстве; `day_of_week` — цифровое обозначение дня недели (0 — понедельник, 6 — воскресенье); `open_time` и `close_time` — время открытия и закрытия; `is_working_day` — признак рабочего дня; `is_bookable` — признак доступности бронирования. Данные расписания используются подсистемой бронирования при генерации 15-минутной временной сетки: слоты за пределами рабочих часов помечаются как недоступные.

Таблица **notifications** содержит уведомления, адресованные пользователям, и сообщения обратной связи от клиентов. Поля: `id_notification` — первичный ключ; `user_id` — получатель; `sender_id` — отправитель (необязательно); `title` и `message` — заголовок и текст; `type` — тип уведомления (`system`, `booking`, `feedback`); `booking_id` — необязательная привязка к конкретному бронированию; `is_read` — флаг прочтения; `created_at` — дата и время создания.

Геометрия планировки — координаты рабочих мест, стены, двери и размеры холста — хранится отдельно от реляционной базы данных в JSON-файле `static/layout.json`. Это решение принято осознанно: редактор планировки часто изменяет координаты объектов, и запись таких изменений в JSON-файл не требует миграций реляционной схемы. Финансовая же история бронирований остаётся в транзакционной PostgreSQL с ACID-гарантиями.

Инициализация базы данных при первом запуске приложения выполняется функцией `init_db()`. Она проверяет существование базы данных, применяет миграции схемы, создаёт все таблицы и загружает начальные данные. Поскольку проект не использует систему миграций Alembic, эволюция схемы реализована функцией `run_migrations()`: при запуске с помощью инспектора SQLAlchemy проверяется наличие необходимых столбцов и при их отсутствии выполняется DDL-команда `ALTER TABLE`, добавляющая недостающий столбец со значением по умолчанию. Фрагмент инициализации представлен на рисунке 36.

```python
# internal/models/seed.py
def init_db(app):
    from internal.config import ensure_database
    ensure_database()
    with app.app_context():
        run_migrations()
        db.create_all()
        init_default_data()
        sync_place_parents_from_layout()
```

## 3.3 Реализация программных модулей системы

Серверная часть построена по многослойной архитектуре: HTTP-обработчики (`internal/handlers/`) принимают запросы Flask и вызывают сервисы (`internal/services/`) с бизнес-правилами. Сервисы обращаются к репозиториям (`internal/repositories/`) и ORM-моделям (`internal/models/`), а также к файлу геометрии `static/layout.json` через пакет `internal/layout/`. Такое разделение ответственности упрощает тестирование компонентов, позволяет повторно использовать бизнес-логику в API и административных формах и снижает связность модулей. Ниже описаны четыре ключевые подсистемы.

### 3.3.1 Подсистема аутентификации и управления доступом

Подсистема аутентификации реализована в пакете `internal/handlers/web/auth.py` совместно с модулем проверки прав `internal/utils/decorators.py`. Подсистема отвечает за регистрацию новых пользователей, вход по паре email–пароль, управление сессией Flask-Login и ролевое разграничение доступа к защищённым маршрутам.

При регистрации система проверяет уникальность адреса электронной почты, нормализует номер телефона через вспомогательный модуль `internal/utils/phone.py` и сохраняет хеш пароля, вычисленный средствами Werkzeug, — открытый пароль в базе данных никогда не хранится. При успешном входе вызывается `login_user` из Flask-Login, создающий зашифрованную сессию в cookie браузера. В зависимости от роли пользователя выполняется перенаправление: администраторы попадают в панель управления, клиенты и менеджеры — в личный кабинет.

Ключевым элементом подсистемы является декоратор `admin_required`, который оборачивает обработчики административных маршрутов. Декоратор проверяет как факт аутентификации, так и наличие роли `admin`. Аналогичные декораторы `staff_required` и `manager_required` реализуют ролевую модель для персонала. Фрагмент декоратора представлен на рисунке 37.

```python
# internal/utils/decorators.py
def admin_required(f):
    @wraps(f)
    @login_required
    def decorated_function(*args, **kwargs):
        if not current_user.is_authenticated or not current_user.is_admin():
            flash('Доступ запрещён. Требуются права администратора.', 'error')
            return redirect(url_for('dashboard'))
        return f(*args, **kwargs)
    return decorated_function
```

Рисунок 37 – Фрагмент кода декоратора проверки прав администратора

Декоратор использует встроенный в Flask-Login `@login_required`, который перенаправляет неаутентифицированных пользователей на страницу входа, и дополняет его проверкой роли. Если пользователь аутентифицирован, но не является администратором, он перенаправляется в личный кабинет с flash-сообщением об ошибке. При успешной проверке управление передаётся исходному обработчику.

### 3.3.2 Подсистема визуализации пространства

Подсистема визуализации пространства обеспечивает отображение интерактивного плана коворкинга и редактирование его геометрии. Подсистема реализована по двухуровневой модели хранения данных: геометрия размещается в файле `static/layout.json`, а бизнес-атрибуты мест — в PostgreSQL. Оба источника объединяются в единый JSON-ответ API по полю `code`, уникальному для каждого рабочего места.

Файл `layout.json` содержит три массива верхнего уровня. Массив `places` описывает рабочие места и зоны: каждый объект содержит поля `code`, `x`, `y`, `width`, `height`, `kind`, `floor` и `location`. Массив `walls` хранит отрезки стен в виде координат двух точек. Массив `doors` описывает дверные проёмы с центром и шириной. Такая структура позволяет отрисовать полный план этажа средствами SVG без обращения к базе данных.

Чтение файла планировки выполняется модулем `internal/layout/store.py`, реализующим кеширование в оперативной памяти: файл читается с диска однократно при первом запросе, а при последующих обращениях возвращается из кеша. Фрагмент модуля хранилища представлен на рисунке 38.

```python
# internal/layout/store.py
_LAYOUT_CACHE = None

def load_layout() -> dict:
    global _LAYOUT_CACHE
    if _LAYOUT_CACHE is None:
        with open(LAYOUT_PATH, 'r', encoding='utf-8') as f:
            _LAYOUT_CACHE = json.load(f)
    return _LAYOUT_CACHE

def invalidate_cache():
    global _LAYOUT_CACHE
    _LAYOUT_CACHE = None
```

Рисунок 38 – Фрагмент кода модуля хранилища планировки

Приведённый фрагмент показывает, что кеш сбрасывается явным вызовом `invalidate_cache()` — это происходит каждый раз при сохранении изменений в редакторе планировки, чтобы следующий запрос к API карты получил актуальные данные.

Серверный API карты реализован в файле `internal/handlers/map/places.py`. При GET-запросе к эндпоинту `/api/places` обработчик загружает геометрию из кеша через `LayoutRepository.load()`, получает актуальные бизнес-атрибуты всех мест из PostgreSQL одним запросом через `PlaceRepository.get_by_codes()` и формирует объединённый массив объектов. Функция `merge_place` соединяет геометрическое описание с записью из базы данных по полю `code`. Фрагмент обработчика представлен на рисунке 39.

```python
# internal/handlers/map/places.py
@app.route('/api/places', methods=['GET'])
def get_places():
    layout        = LayoutRepository.load()
    layout_places = layout.get('places', [])
    codes         = {p['code'] for p in layout_places}
    db_places     = PlaceRepository.get_by_codes(codes)
    result = [merge_place(lp, db_places) for lp in layout_places]
    return jsonify({
        'places': result,
        'walls':  layout.get('walls', []),
        'doors':  layout.get('doors', []),
    })
```

### 3.3.3 Подсистема бронирования

Подсистема бронирования реализована в файле `internal/services/booking_service.py` и содержит всю бизнес-логику, связанную с проверкой доступности рабочих мест и формированием временной сетки. Бронирование работает с 15-минутной сеткой, минимальный интервал составляет 30 минут (два слота), максимальный — 8 часов. Поддерживаются три типа тарифов: почасовой (`hourly`), недельный (`weekly`) и месячный (`monthly`).

Ключевым элементом подсистемы является класс `TimeSlot`, представляющий один 15-минутный интервал. Каждый слот хранит время начала и окончания, количество занятых мест, общую вместимость и список идентификаторов пересекающихся бронирований. На основании соотношения занятых и доступных мест вычисляется статус слота: `free` — полностью свободно, `partial` — частично занято и можно добавить посетителей, `full` — место занято целиком. Фрагмент класса TimeSlot представлен на рисунке 42.

```python
# internal/services/booking_service.py
@dataclass
class TimeSlot:
    start_time:     time
    end_time:       time
    occupied_seats: int
    capacity:       int
    bookings:       List[int]

    @property
    def status(self) -> TimeSlotStatus:
        if self.occupied_seats == 0:
            return TimeSlotStatus.FREE
        elif self.occupied_seats < self.capacity:
            return TimeSlotStatus.PARTIAL
        return TimeSlotStatus.FULL
```

Рисунок 42 – Фрагмент кода класса TimeSlot

Метод `get_timegrid_for_place` генерирует полную сетку слотов для указанного места на заданную дату, опираясь на расписание коворкинга из таблицы `coworking_schedules` и существующие бронирования из таблицы `bookings`. Слоты за пределами рабочих часов исключаются из ответа. При создании нового бронирования сервис проверяет, что ни один из запрошенных слотов не является полностью занятым для указанного количества посетителей — это исключает конфликт интервалов при параллельных запросах нескольких пользователей.

Стоимость бронирования рассчитывается функцией `_calc_booking_total()` на основе типа тарифа, длительности интервала и числа посетителей. Для почасового тарифа цена масштабируется пропорционально числу 15-минутных слотов от часовой ставки. Для недельного и месячного тарифов цена умножается на количество посетителей, что позволяет отражать совместное использование переговорных комнат. Фрагмент расчёта стоимости представлен на рисунке 43.

```python
# internal/handlers/booking/api.py
def _calc_booking_total(place, tariff_type, start_time,
                        end_time, people_count) -> int:
    tariff = CategoryTariff.query.filter_by(
        category_id=place.category_id,
        tariff_type=tariff_type, active=True
    ).first()
    if not tariff:
        return 0
    if tariff_type == 'hourly':
        slots  = _count_slots(start_time, end_time)
        price  = int(tariff.price * slots / 4)   # 4 слота = 1 час
    else:
        price  = int(tariff.price * people_count)
    return price
```

Рисунок 43 – Фрагмент кода расчёта стоимости бронирования

Подсистема также поддерживает применение абонементов: перед созданием бронирования проверяется наличие у клиента активного абонемента, охватывающего запрошенный тип места. При наличии подходящего абонемента бронь создаётся бесплатно, а из остатка абонементных часов списывается соответствующее количество.

В левой колонке расположена палитра шаблонов мест, в центральной — SVG-холст с сеткой и размещёнными объектами, в правой — панель свойств выбранного объекта. В верхней панели доступны инструменты размещения стен, дверей и рабочих мест, сохранения планировки и переключения этажей. После нажатия «Сохранить» изменения записываются в `layout.json`, кеш планировки сбрасывается и выполняется синхронизация с PostgreSQL. Экран редактора планировки представлен на рисунке 16.

Рисунок 16 – Экран редактора планировки

Ниже выводятся карточки ключевых показателей и два графика в одном ряду: линейная диаграмма «Посетители по дням» и столбчатая диаграмма «Пиковые часы».
